# 일정 조율의 본질적 문제 정의

> 이 문서는 솔루션 설계 이전, **"진짜 문제가 무엇인가"**를 정의하는 리서치 단계 산출물입니다. 화면/컴포넌트/hex 같은 디자인 구체화는 다음 단계로 미루고, 여기서는 **문제의 본질**에만 집중합니다.

## 들어가며 — 우리가 풀려는 건 "일정 조율"이 아니다

표면적으로 6명이 1시간을 맞추는 건 단순한 **달력 교집합** 문제처럼 보입니다.
하지만 카카오톡·이메일에서 일정을 잡아본 사람은 압니다. 진짜 고통은 "빈 시간을 찾는 것"이 아니라 **"누가 언제 되는지를 묻고, 모으고, 해석하고, 결정하고, 통보하는" 전 과정의 인지·정서 비용**에 있습니다.

아래 3가지가 그 본질입니다.

---

## 문제 1 — 응답을 모으는 사람이 모든 비용을 떠안는다

**한 줄 요약: 일정 조율은 6명이 나눠야 할 부담을 주최자 1명이 통째로 떠안는 구조다.**

### 무엇이 일어나는가
카톡방에 "다음 주 언제 1시간 돼요?"를 던지면, 답은 비동기로·자유 텍스트로·서로 다른 형식으로 흩어져 돌아옵니다.

- "저는 화수 빼고 다 돼요"
- "오전만 가능"
- "목요일은 외근이라 안 돼요"
- (그리고 2명은 읽씹)

### 왜 문제인가 (심리·행동 관점)
- **번역 노동**: 주최자는 6개의 자연어 응답을 머릿속에서 격자(요일×시간)로 변환해야 합니다. 이건 인간이 가장 못하는 작업 — *비정형 정보의 교집합 연산*입니다.
- **책임의 비대칭**: 응답자는 "한마디 던지면 끝"이지만, 주최자는 모든 누락·충돌·재확인의 책임을 집니다. 이 불공평함이 **"내가 굳이 총대 메야 하나"**라는 회피 심리를 낳고, 회의 자체가 미뤄집니다.
- **종결 불안**: "다 모았나? 빠진 사람 없나?"를 계속 확인해야 하는 미완결 상태가 인지적으로 가장 피로합니다(자이가르닉 효과 — 끝나지 않은 일이 머리를 떠나지 않음).

---

## 문제 2 — 가능/불가능만 보일 뿐, "최선"은 보이지 않는다

**한 줄 요약: 기존 도구는 빈 시간은 알려줘도, 6명에게 가장 덜 미안한 시간이 언제인지는 알려주지 않는다.**

### 무엇이 일어나는가
설령 모두의 일정을 모았다 해도, 현실의 답은 "모두 100% 가능한 시간"이 아닌 경우가 대부분입니다. 결국 누군가는 양보해야 하고, **그 양보를 누구에게 시킬지**가 진짜 결정입니다.

### 왜 문제인가 (심리·행동 관점)
- **결정의 정당화 부담**: "왜 하필 그 시간이냐"에 답할 논리가 없으면, 주최자는 결정을 못 내립니다. 빈칸이 아니라 **"이게 최선인 이유"**가 필요합니다.
- **사회적 부채감**: 누군가를 불가능한 시간에 부르는 건 관계 비용을 발생시킵니다. 그래서 사람들은 차라리 회의를 미루는 — 즉 **갈등 회피로서의 무결정**을 택합니다.
- **모든 가능성이 동등해 보이는 함정**: 빈 시간이 5개 나오면, 인간은 오히려 못 고릅니다(선택 과부하). "추천 1순위"가 없으면 결정은 다시 정체됩니다.

> 핵심 통찰: 사람들이 원하는 건 *선택지*가 아니라 *근거 있는 추천*입니다.

---

## 문제 3 — 확정과 통보 사이에서 합의가 증발한다

**한 줄 요약: "그럼 목요일 3시로 하죠"라고 말한 순간과 6명이 실제로 그 시간을 비워두는 순간 사이에, 합의가 새어 나간다.**

### 무엇이 일어나는가
카톡에서 "목 3시요!" 한마디로 끝난 회의는, 며칠 뒤 "어? 그거 확정이었어요?" "캘린더에 안 넣었는데"로 무너집니다. 결정과 실행이 분리돼 있기 때문입니다.

### 왜 문제인가 (심리·행동 관점)
- **구두 합의의 휘발성**: 채팅 스크롤 위로 올라간 결정은 존재하지 않는 것과 같습니다. **단일한 "확정된 사실"의 출처(single source of truth)가 없습니다.**
- **책임 분산의 착각**: 모두가 "누군가 캘린더에 넣겠지"라고 생각하면 아무도 안 넣습니다(방관자 효과).
- **Full-Cycle의 단절**: Trigger(언제 돼요?) → 응답 → 결정까지는 어찌어찌 가도, **확정 → 캘린더 등록 → 리마인드**가 끊깁니다. 토스 관점에서 이건 "오너십이 끝까지 안 간" 미완성 경험입니다.

---

## 제약 조건이 문제를 어떻게 심화시키는가

세 가지 조건은 각각 위 문제를 **증폭**시킵니다. 단순히 "조건이 추가됐다"가 아니라, **문제의 본질을 더 날카롭게 만드는 지렛대**입니다.

| 조건 | 어떤 문제를 심화시키는가 | 심화의 메커니즘 |
|---|---|---|
| **필수/선택 참석자 구분** | 문제 2 (최선이 안 보임) | 단순 교집합이 **가중 교집합**으로 바뀝니다. "6명 다 되는 시간"이 아니라 "필수 4명은 반드시, 선택 2명은 가능하면"이라는 비대칭 계산이 필요해, 사람 머리로는 거의 불가능. 추천 알고리즘의 근거가 더 절실해집니다. |
| **특정 요일 외근** | 문제 1 (응답 수집) | 외근은 "그날 통째로 불가"라는 **블록 단위 제약**입니다. 자유 텍스트로는 "수요일은 외근 많아요(=확실히 안 되나? 가끔 되나?)"처럼 모호하게 표현돼, 주최자의 번역 노동과 재확인 비용을 키웁니다. |
| **점심 직후 기피** | 문제 2 (최선이 안 보임) | 이건 "불가능"이 아니라 **"가능하지만 싫음" = 선호 가중치**입니다. 기존 도구는 가능/불가능 이분법만 다루므로, 이 회색지대를 표현할 방법이 없습니다. 결국 당사자는 말 안 하고 참거나, 뒤늦게 불만을 표합니다(억눌린 비용). |

### 종합 — 제약이 만드는 진짜 난이도
세 조건이 합쳐지면, 문제는 단순한 **boolean 교집합**에서 **다층 가중 최적화**로 격상됩니다:

> "필수 참석자 4명은 무조건 가능 + 외근 요일은 제외 + 점심 직후는 감점, 이 조건에서 6명 전체의 미안함 총합이 가장 작은 1시간은?"

이 계산을 사람이 카톡으로 한다는 것 자체가 비현실적입니다.
**바로 여기에 토스가 단순화로 개입할 지점이 있습니다.**

## 다음 단계로 넘기는 핵심 정의 (Solution이 풀어야 할 것)

리서처로서 다음 단계 설계팀에 넘기는 **문제 정의 명제**는 다음 3가지입니다.

1. **수집의 비대칭을 없애라** — 응답을 "텍스트"가 아니라 "구조화된 입력"으로 받아, 주최자의 번역 노동을 0으로 만든다.
2. **가능 대신 최선을 보여줘라** — 가능/불가능/선호 3단계와 필수/선택 가중치를 반영해, "이 시간이 최선인 이유"를 근거와 함께 단 하나로 추천한다.
3. **결정을 사실로 만들어라** — 확정 즉시 6명의 캘린더 등록과 알림까지 자동으로 이어, 합의가 휘발되지 않게 한다.

> 이 3개 명제가 이후 **솔루션 설계 → 화면/컴포넌트 → Figma Make 프롬프트**로 구체화될 토대입니다. 필요하시면 다음 단계로 솔루션 설계와 모바일/웹 화면 사양(컴포넌트·카피·hex·간격)까지 이어서 진행하겠습니다.

## 1. 기능 정의 (Feature Set)

## 2. Full-Cycle 유저 플로우

```
[Trigger]
주최자: "회의 만들기" → 제목·길이(1시간 고정)·기간(다음 주) 입력
       → 참석자 6명 초대 + 각자 필수/선택 토글 (F7)
       → 초대 링크/푸시 발송
        ┌─ 모바일 진입: 푸시 알림 탭 → 앱 바로 응답 화면
        └─ 웹 진입: 링크 클릭 → 로그인 없이 게스트 응답 가능

[응답 수집]
참석자: 격자 탭으로 가능/선호/불가 입력 (F1)
       외근 요일은 헤더 길게눌러 일괄 차단 (F3)
       → "제출" → 트래커 +1 (F6)
주최자: 미응답자에게 콕 찌르기 (F4)

[추천]
6명(또는 필수 전원) 응답 완료 → 베스트 타임 엔진 가동 (F2)
       → 1순위 + 차순위 2개를 근거와 함께 제시

[확정]
주최자: 추천 카드 "이 시간으로 확정" 원탭 (F5)

[알림]
6명 전원에게 확정 푸시 + 캘린더 자동 등록
       → "회의가 확정됐어요 🎉 목 15:00–16:00"
```

**모바일 vs 웹 차이**
- 모바일: 푸시 → 단일 컬럼 응답에 집중. 빠른 응답이 핵심.
- 웹: 주최자가 6명 격자를 동시에 펼쳐보는 **대시보드형 멀티 컬럼**. 정보 밀도 높음.

## 3. 핵심 화면 설계

### 공통 디자인 토큰
- **Primary** `#3182F6` (토스 블루) / **선호** `#3182F6` / **가능** `#E5F0FF` / **불가** `#F2F4F6` / **경고/감점** `#FF8A00`
- **Text** 강 `#191F28` / 약 `#8B95A1` / 배경 `#FFFFFF`, 섹션 `#F9FAFB`

- 모서리 반경 16px (카드) · 12px (버튼) / 기본 간격 16px / 버튼 높이 56px

---

### 화면 ① 응답 입력 (격자) — 모바일 주력

- **목적**: 참석자가 가장 빠르게 가능/선호/불가를 입력
- **플랫폼**: 모바일 우선

**UI 구성 (위→아래)**
1. 상단 바: ← / "기획 리뷰 회의" / 진행률 점 (3/6)
2. 안내 배너: "탭할수록 바뀌어요 · ⬜가능 → 🔵선호 → ✕불가"
3. 요일 헤더 (월~금) — 길게 누르면 "OO요일 종일 불가" 처리 (F3)
4. 시간 격자 (09–18시, 1시간 셀) — 점심직후(13시) 칸에 `🍽️` 미세 아이콘
5. 하단 고정 CTA: **"응답 제출하기"** (56px, `#3182F6`)

**카피**
- 헤더: "다음 주, 언제 1시간 되세요?"
- 빈 상태: "되는 시간을 탭해서 알려주세요"
- 완료 토스트: "응답 완료! 주최자가 시간을 정하면 알려드릴게요"

**상태**
- 기본: 전체 ⬜
- 빈 상태: 아무것도 안 누름 → CTA 비활성(`#D1D6DB`)
- 로딩: 제출 시 버튼 스피너
- 에러: "전송에 실패했어요. 다시 시도해주세요" (`#FF5247`)
- 완료: 격자 dim + ✓ 오버레이

**모바일 vs 웹**: 모바일=세로 스크롤 단일 격자 / 웹=요일 가로 전체 노출, 마우스 드래그로 다중 셀 선택.

**설계 의도**: 탭 순환으로 3단계를 한 손가락으로 표현하면, 사용자는 "선택지 고민" 없이 칠하듯 입력하여 응답 이탈이 줄어들 것이다.

---

### 화면 ② 주최자 현황 대시보드 — 웹 주력

- **목적**: 6명 응답을 한눈에 보고 미응답자를 찌르기
- **플랫폼**: 웹 우선(정보 밀도 높음)

**UI 구성 (위→아래)**
1. 헤더: 회의명 + "다음 주까지" + 진행률 바
2. 참석자 칩 6개: 아바타 + 이름 + [필수]/[선택] 배지 + 응답✓/대기⏳
   - 대기 칩에 **"콕 찌르기"** 버튼 (F4)
3. 통합 히트맵 격자: 셀이 진할수록 다수 가능(흰→`#E5F0FF`→`#3182F6`)
4. 우측 사이드: 베스트 타임 추천 카드 (다음 화면)

**카피**
- "아직 2명이 응답하지 않았어요"
- 찌르기 후 토스트: "김선호님에게 알림을 보냈어요 👉"

**상태**
- 빈 상태(응답 0): "초대한 동료들의 응답을 기다리고 있어요"
- 로딩: 칩 스켈레톤
- 완료(6/6): 추천 카드 활성화 + "이제 시간을 정할 수 있어요"

**모바일 vs 웹**: 웹=히트맵+사이드 추천 2단 / 모바일=탭 전환(현황↔추천), 칩은 가로 스크롤.

**설계 의도**: 미응답자를 회색으로 분리·찌르기 버튼을 붙이면, 주최자는 추적을 시스템에 위임하여 정서적 부채를 덜 것이다.

---

### 화면 ③ 베스트 타임 추천 — 모바일·웹 공통 핵심

- **목적**: 근거 있는 단 하나의 추천으로 결정 정체 해소

**UI 구성 (위→아래)**
1. 헤더: "가장 미안함이 적은 시간을 찾았어요"
2. **1순위 카드** (강조, 그림자 + `#3182F6` 테두리):
   - "목요일 15:00 – 16:00"
   - 적합도 게이지 92% + "필수 4명 전원 가능 · 선호 2명"
   - 근거 칩: ✅필수 전원 / 💙선호 다수 / ⚠️없음
   - CTA **"이 시간으로 확정하기"**
3. 차순위 카드 2개(축소): "화 10:00 (필수 전원·선호 1)" 등
4. 하단 텍스트 링크: "직접 다른 시간 고르기"

**카피**
- 1순위 뱃지: "1순위 추천"
- 감점 사유 노출: "수요일은 외근으로 제외했어요"

**상태**
- 로딩: "6명의 일정을 계산하고 있어요…" (1.5s 애니메이션)
- 에러(공통 시간 없음): "모두 되는 1시간이 없어요. 필수 인원만으로 추천할까요?" → 대안 버튼
- 완료: 확정 후 ③ → ④ 전환

**모바일 vs 웹**: 모바일=1순위 풀폭+차순위 세로 / 웹=3개 카드 가로 정렬.

**설계 의도**: "92% · 근거 칩"으로 추천 이유를 시각화하면, 주최자는 결정을 정당화할 논리를 얻어 즉시 확정할 것이다.

---

### 화면 ④ 확정 완료 — 공통

- **목적**: 합의를 "사실"로 고정, Full-Cycle 종결

**UI 구성**
1. 성공 일러스트 + 헤더 "회의가 확정됐어요 🎉"
2. 확정 카드: "목 15:00–16:00 · 참석 6명"
3. **"내 캘린더에 추가"** (자동 등록 완료 표시 ✓)
4. "참석자 모두에게 알림을 보냈어요" 안내
5. 보조: "회의 정보 공유하기"

**카피**: "6명의 캘린더에 자동으로 등록했어요. 더 이상 확인하지 않아도 돼요."

**상태**: 로딩(등록 중 스피너) / 완료(✓) / 에러("캘린더 연동 실패 — 수동 추가하기")

**설계 의도**: 확정 즉시 자동 등록을 명시하면, 사용자는 "누가 캘린더 넣지?" 방관자 효과 없이 종결감을 느낄 것이다.

## 4. 필수/선택 참석자 가중치 — 시각적 표현

| 구분 | 크기 | 색 | 위치/형태 |
|---|---|---|---|
| **필수** | 아바타 **40px**, 테두리 2px | 링 `#3182F6` (블루) | 칩 좌측 우선 정렬, "필수" 배지(블루 pill) |
| **선택** | 아바타 **32px**, 테두리 1px | 링 `#8B95A1` (그레이) | 필수 뒤 배치, "선택" 배지(그레이 pill) |

**격자/히트맵 반영**
- 셀 채도 계산 시 필수 가중치 ×2, 선택 ×1 → **필수가 안 되는 칸은 시각적으로 강하게 빠짐**(연한 회색 빗금).
- 추천 카드 근거 칩에서 필수는 ✅(굵게), 선호는 💙(보조)로 위계 구분.

**설계 의도**: 필수를 더 크고 진한 블루로, 선택을 작고 흐린 그레이로 표현하면, 사용자는 학습 없이 "누구를 먼저 맞춰야 하는지"를 직관적으로 인지할 것이다.

## 페르소나 1️⃣ : 주최자 — 회의를 소집하고 다음 주까지 빨리 확정해야 하는 사람

**첫인상**
"오, 이건 내가 직접 안 물어봐도 되겠네. 근데 정말 다 모여야 하나?" 
화면을 보자마자 6명 칩과 진행률이 바로 보여서 "아, 시스템이 나 대신 추적해주는구나"라는 안도감이 든다.

**막히는 지점**
참석자를 "필수/선택"으로 나눠야 한다는 걸 처음에 안다. 그런데 확인 버튼이 어디 있는지 한번 헤맨다.
초대 링크를 발송하는 과정이 명확하지 않음 — "발송됐나? 언제 답장 올까?" 이 불안이 계속 남아있다.

**오해할 부분**
"필수 4명 전원 가능"이라는 근거 칩을 봤을 때, "선택 2명은 못 하는 건가?"라고 오해할 수 있다.
실제론 "필수만 계산했고, 선택은 가능하면 포함"인데, 근거 표현이 사실 불명확하다.

**감정 반응**
- 응답 완료 전: 불안함 (혹시 모두가 응답할까? 일정이 안 나올까?)
- "베스트 타임" 카드 본 순간: 엄청 안심 (계산을 내가 안 해도 되네!)
- 확정 후: 만족감 (이제 캘린더 알림까지 자동이라고? 정말?)

**한 줄 제안**
**초대 직후 바로 "2명이 아직 응답 안 했어요 → 지금 재촉할까?"라는 능동적 넛지를 주세요.**
지금은 내가 콕 찌르기를 능동적으로 눌러야 하는데, 자동으로 "답장 기다리는 중" 알림이 오면 정신 덜 쓸 것 같아요.

---

## 페르소나 2️⃣ : 점심 직후 기피자 — 식곤증 때문에 점심 직후 시간대를 피하고 싶음

**첫인상**
"응답 화면 들어가는데 13시 칸에 🍽️ 표시가 있네? 아, 이게 나를 위한 거구나."
다른 앱에선 이런 세심함이 없었는데, 뭔가 나를 이해하는 느낌이 든다.

**막히는 지점**
그런데 13시가 "피하는 게 낫다"는 뜻인지 "하면 안 된다"는 뜻인지 명확하지 않다.
탭했을 때 강제로 불가(✕)가 되는 건 아니고, 나는 "선호하지 않음" 정도의 의사를 표현할 수 있나?
실제로 탭하면 ○ → ◎ → ✕ 순환인데, **중간에 "회색 상태(선호 아님)"가 없어서**, 
내가 할 수 있긴 하지만 좋지 않다는 세밀한 의사를 전달할 수 없다.

**오해할 부분**
13시에 🍽️가 있으니까 "13시는 남의 제약이구나"라고 생각하고, 내 진짜 기피 시간대인 14~15시는 어떻게 표현하지?
시스템이 내 개인 선호도를 미리 저장할 수 없으니, 매번 손으로 칠해야 한다.

**감정 반응**
- 초기: 기대감 (내 시간대를 아는 시스템이 드물어서 좋음)
- 응답 중: 약간의 불안 (13시를 안 누르려고 집중하다가 실수할까?)
- 확정 후: 약간의 의심 (이 시간 안 좋은데 내가 선호도를 제대로 전달했나?)

**한 줄 제안**
**4단계 상태를 만들어주세요: ○가능 → ◔선호하지 않음 → ◎선호 → ✕불가.**
그럼 나는 13시를 "◔"으로 표현해서 "이 시간도 되지만 좋진 않아"라는 의사를 정확히 전달할 수 있어요.

---

## 페르소나 3️⃣ : 외근 잦은 동료 — 특정 요일에 외근이 많아 일정 충돌이 잦음

**첫인상**
"수요일 헤더에 🧳 아이콘 있네? 아, 길게 누르면 종일 불가 처리되는 거구나."
외근이 많은 내 입장에서는 이게 정말 꿈같은 기능이다.
시간 하나하나 ✕ 칠할 필요 없이 요일을 한 번에 블록할 수 있다니.

**막히는 지점**
근데 수요일이 "항상" 외근인 건 아니고, 보통 오전에 외근이고 오후는 돌아오는 경우가 많다.
화면에선 요일 단위 블록만 보이는데, 시간 단위로 세분할 수는 없나?
장시간 누르기(long press)를 실제로 했을 때 feedback이 명확한지도 모르겠다.

**오해할 부분**
"수요일 헤더 길게누름 → 종일 외근"이라는 인터랙션이 직관적이지 않을 수도 있다.
첫 사용자는 이게 누르는 건지 드래그하는 건지 길게 잡는 건지 헷갈릴 것 같다.
시각적으로 "누르면 뭔가 일어난다"는 신호(예: 장시간 누르기 힌트)가 없다.

**감정 반응**
- 초기: 호기심 (외근 관련 기능이 있다니!)
- 상호작용 중: 약간의 답답함 (하루 중 일부만 외근인데 전체 요일을 못 쓰게 되는 건 아닐까?)
- 확정 후: 안심 (어쨌든 시스템이 내 외근을 반영했네)

**한 줄 제안**
**외근 시 "수요일 09~12시만 외근, 14시부터는 가능" 같은 부분 블록 기능을 추가하거나,
최소한 "수요일 종일 외근" 버튼을 명시적으로 보여주세요 (길게누르기보다는 탭 후 확인 팝업).**

---

## 페르소나 4️⃣ : 선택 참석자 — 필수가 아니라 우선순위가 낮고, "내 사정이 반영될까" 눈치를 봄

**첫인상**
"[선택] 배지가 내 칩에 붙어있네. 아, 나는 필수가 아니구나."
그 순간부터 약간의 위축감이 든다. 내가 대답해도 상관없는 건 아닐까?

**막히는 지점**
대시보드에서 필수 참석자 4명이 큰 아바타(40px)로 우선 배치되고, 나는 작은 아바타(32px)로 뒤에 밀려있다.
응답 화면에서도 내가 특별히 다르게 표현되거나 "너는 선택이니까 최소 1명만 응답하면 돼" 같은 안내가 없다.
그래서 나는 응답해야 하는지 말아야 하는지 약간 애매한 기분이 든다.

**오해할 부분**
베스트 타임 추천 카드에 "필수 4명 전원 가능 · 선호 2명"이라고 나오는데, 
"선호 2명"이 나를 포함한다는 것 자체를 모를 수 있다.
혹시 필수 4명만으로 이미 결정되고, 내 응답은 참고만 하는 건 아닐까?

**감정 반응**
- 응답 화면 진입: 미안함 (내가 응답하는 게 맞나?)
- 응답 중: 신경 쓰임 (혹시 내가 "선호"로 표시한 시간이 무시되지는 않을까?)
- 확정 후: 의외의 안심 (나도 추천 카드에 "선호 2명"으로 반영되었네!)

**한 줄 제안**
**선택 참석자에게 "당신은 필수 참석자가 아니지만, 가능한 범위 내에서 모두를 배려하는 시간을 찾고 있어요"라는 명시적 메시지를 주세요.**
그럼 나도 "아, 내가 응답할 이유가 있구나"라고 느껴서 성실하게 응답할 것 같아요.

## 페르소나 5️⃣ : 바쁜 필수 참석자(리더) — 시간이 없고 빠른 의사결정을 선호함

**첫인상**
"격자 화면 떴는데, 내가 이걸 다 칠해야 하나? 정신없네."
응답 화면의 요일×시간 격자가 눈에 들어왔을 때, "이 모든 칸에 응답해야 하나?"라는 부담감이 즉각 든다.

**막히는 지점**
내가 바쁜 사람인데, 격자를 모두 탭해야 한다는 게 너무 느리다.
드래그로 다중선택을 할 수 있다고 했지만 모바일에서는 그게 잘 먹히지 않는다.
결국 나는 "음... 딱 3개 시간만 되는데" 하고 손가락으로 3개를 하나하나 찍는다 — 3초, 5초...

**오해할 부분**
"빨리 답장해야겠다"는 생각에 격자를 성급하게 탭하다가, 
내가 실제로 가능한 시간을 정확히 표현했는지 확신이 없다.
응답 제출 직전에 "이거 맞나?" 하고 다시 보니까 1개가 잘못 눌러져 있다.

**감정 반응**
- 응답 화면 진입: 귀찮음 (또 이런 거 해야 해?)
- 응답 중: 초조함 (빨리 끝내고 싶은데 계속 칸을 확인하게 됨)
- 제출 후: 약간의 불안 (정말 내가 정확하게 표시했나?)

**한 줄 제안**
**"내가 가능한 시간만 콕 3개 선택" → 자동으로 응답 완료라는 "빠른 응답 모드"를 만들어주세요.**
아니면 최소한 모바일에서 드래그 다중선택이 정말 부드럽게 작동하도록 최적화해주세요.
그럼 나는 3초 안에 응답을 끝낼 수 있을 것 같아요.

## 페르소나 6️⃣ : 신규입사자/디지털 약자 — 학습 없이 직관적으로 쓸 수 있어야 함

**첫인상**
"와, 앱이 깔끔하긴 한데... 이게 뭔 앱이고 내가 뭘 해야 하는 거지?"
링크를 타고 들어왔는데, 별도의 가이드나 온보딩이 없다.
화면 맨 위에 "← 기획 리뷰 회의"라는 제목만 있고, "내가 뭘 해야 해?"라는 게 한 줄도 없다.

**막히는 지점**
격자 화면이 떴는데, 요일과 시간이 무엇인지는 분명하지만,
"탭할수록 바뀐다"는 설명은 작은 글씨로만 있고, 혹시 내가 이해를 잘못한 건 아닐까 불안하다.
첫 탭을 했을 때 ○가 ◎로 바뀌는 건 시각적으로 명확하지만, 
"이게 뭘 의미하는 건지"는 설명이 부족하다.
(○=가능, ◎=선호, ✕=불가라는 게 진짜 이것 맞나?)

**오해할 부분**
"다음 주, 언제 1시간 되세요?"라는 헤더 문구가 좀 딱딱하게 느껴진다.
초보자는 "1시간"이 정확히 무엇인지 (회의가 1시간이구나? 나는 1시간을 연속으로 내줘야 하나?) 
문맥상 이해하지만, 한 번 더 명시적으로 설명되면 좋을 것 같다.

점심직후 🍽️ 아이콘은 예쁘지만, "이게 왜 있는 거야?"라고 물어볼 가능성이 있다.
아이콘만으로는 "누군가의 기피 시간"이라는 게 명확하지 않다.

**감정 반응**
- 진입: 혼란 (이게 뭐지? 내가 뭘 해야 하지?)
- 응답 중: 신중함 (혹시 틀리면 어쩌지?)
- 완료 후: 안심 감 (어라, 생각보다 쉬운데? 그냥 칠하고 제출하면 되네)

**한 줄 제안**
**화면 진입 직후 2초짜리 "동작 가이드 팝업" (○는 가능, ◎는 선호, ✕는 불가예요. 
탭해서 알려주세요!)을 한 번 보여주고, "닫기" 또는 "3초 후 자동 닫기"로 처리해주세요.**
그럼 신입도 자신감 있게 응답할 것 같아요.

## [공통 발견 사항] — 반복적으로 나타난 핵심 사용성 이슈

### 이슈 1️⃣ : 4단계 선호도의 필요성 — 현재 3단계는 "회색지대"를 놓친다

**페르소나별 증거**
- 페르소나 2 (점심 직후 기피자): 13시를 피하고 싶지만, ○(가능) 또는 ✕(불가) 중 선택을 강요받음
- 페르소나 1 (주최자): "최선의 시간"을 찾을 때, "할 수는 있지만 좋지 않은 시간"의 정보 부족
- 페르소나 5 (바쁜 리더): 응답할 때 "이 시간도 되는데 좋지는 않아"를 표현할 방법이 없음

**핵심 문제**  
현재 시스템은 ○가능 → ◎선호 → ✕불가로 설계되었으나, 실제로는 **"할 수 있지만 별로인" 회색지대**가 매우 흔하다.
특히 점심직후, 저녁 늦은 시간, 또는 외근 가능성이 높은 시간대는 "완전히 불가는 아니지만 피하고 싶다"는 상태다.
이를 표현할 수 없으면, 주최자는 추천 알고리즘을 신뢰할 수 없다.

**해결책**  
**4단계 상태를 도입**: ○가능 → ◔회피(회색 음영) → ◎선호 → ✕불가.
시각적으로는 회피를 진하고 연한 블루의 중간 톤(예: `#B0D4FF`)으로, 아이콘은 반쪽 채운 원(◔)으로 표현.
추천 알고리즘은 회피를 "감점 가중치"로 반영하여, 불가는 아니지만 덜 바람직한 시간을 자동으로 피함.

---

### 이슈 2️⃣ : 외근/부분 차단의 세밀한 표현 필요 — 현재 "요일 전체" 블록만으로는 부족

**페르소나별 증거**
- 페르소나 3 (외근 잦은 동료): 수요일이 "항상" 외근은 아니고, 보통 오전만 외근인데 요일 전체를 블록할 수 없음
- 페르소나 1 (주최자): 외근이 "대략" 어느 시간인지 알지만, 정확히 표현할 방법이 없음

**핵심 문제**  
현재 설계는 헤더 길게누르기로 "수요일 종일 외근" 처리만 가능하다.
하지만 실제 외근은 시간대가 다양하다:
- "수요일 오전 9~12시만 외근, 14시부터는 가능"
- "목요일은 가끔 외근이 있을 수 있음"

단순 요일 블록만으로는 이런 세밀함을 표현할 수 없고, 
결국 각 칸을 하나하나 ✕로 칠해야 하는데 이것은 UX 저하다.

**해결책**  
**2단계 인터랙션**:
1. 헤더를 **탭**하면 "이 요일 상태 선택" 팝업 → ○전일 가능 / ◔부분 외근 / ✕종일 외근
2. ◔부분 외근을 선택하면, 그 요일의 시간 격자만 펼쳐져서 "09~12시는 ✕, 나머지는 ○" 식으로 세밀하게 입력 가능.

또는 더 간단하게: **격자 내 드래그로 시간 범위를 선택해서 일괄 ✕ 처리** (예: 09시부터 12시까지 드래그 → 3개 셀이 한 번에 ✕).

### 이슈 3️⃣ : 초보자/신입을 위한 명시적 가이드 및 필드 라벨 부족

**페르소나별 증거**
- 페르소나 6 (신규입사자): 앱 진입 후 "내가 뭘 해야 하지?"라는 의문, 🍽️ 아이콘 의미 불명확
- 페르소나 5 (바쁜 리더): "탭할수록 바뀐다"는 설명이 작아서 처음 한두 번은 헷갈림
- 페르소나 1 (주최자): 필수/선택 토글 위치 및 동작 불명확

**핵심 문제**  
화면이 깔끔하다는 게 장점이지만, 동시에 **초보자를 위한 진입 가이드가 거의 없다**.
특히 모바일 앱으로는 온보딩이나 팝업 가이드가 없으면, 사용자는:
- "○ ◎ ✕의 정의가 뭔지" 추측만 함
- 🍽️ 아이콘이 "내 기피 시간"을 의미한다는 걸 모를 수도 있음
- 초대 링크 발송이 정말 완료됐는지 확신이 없음
- 수요일 헤더 길게누르기의 정확한 인터랙션을 모름

**해결책**  
**1단계: 초대 후 첫 진입 가이드**
- 응답 화면 로드 시 1회성 팝업: "탭할수록 바뀌어요 ○가능 → ◎선호 → ✕불가"
  (또는 헬퍼 텍스트로 항상 보이도록)
- 🍽️ 마우스 오버/롱프레스 시 툴팁: "점심시간 이후라 피하면 좋아요"

**2단계: 명시적 필드 라벨**
- 현재 헤더만 있는 것 말고, 각 영역에 라벨 추가:
  - "다음 주, 언제 1시간 되세요?" (현재) → 하단에 작은 설명: "(1시간 회의입니다)"
  - 요일 헤더 우측에 "ℹ️" 아이콘 → "길게누르면 종일 외근 처리" 팁

**3단계: 주최자 초대 후 확인**
- 초대 링크 발송 후: "✓ 6명이 초대됐어요. 첫 응답이 오면 알려드릴게요" 토스트
  → 링크가 정말 발송된 것을 시각적으로 확인시킴

## 종합 결론

이 6명의 페르소나를 통해 드러난 핵심은:

1. **"가능/불가"만으로는 충분하지 않다** — 사람들은 회피, 선호, 필수라는 다층적 의사를 갖고 있고, 이를 모두 수용하는 4단계 체계가 필요하다.

2. **"외근은 정확해야 한다"** — 신뢰의 기초는 "내 사정이 정말 반영되는가"에 있으므로, 블록 단위 차단과 시간 단위 세밀화를 모두 지원해야 한다.

3. **"설명 없이는 사용 불가"** — 아무리 깔끔한 디자인이어도 초보자는 가이드가 없으면 진입 자체를 망설인다. 첫 응답을 완료할 때까지의 마찰을 극도로 줄여야 한다.

**이 세 가지를 개선하면, 6명의 페르소나가 모두 "이 도구는 내 사정을 이해한다"고 느끼게 될 것이다.**

## 기준 1 — 본질적인 문제 정의

### 판정: **PASS** (9/10)

**근거 (강점)**
- Agent A는 "달력 교집합" 같은 표면 현상을 명확히 부정하고, **수집의 비대칭(자이가르닉·책임 비대칭) → 최선 부재(선택 과부하·사회적 부채) → 합의 휘발(방관자 효과·single source of truth 부재)**라는 3층 심리/행동 허들을 정확히 짚었습니다. 이건 "기능 리스트"가 아니라 "왜 사람이 회의를 미루는가"라는 회피 심리까지 파고든 정의입니다.
- 특히 제약 조건을 "추가 요구사항"이 아니라 **문제의 본질을 증폭시키는 지렛대**로 재해석한 표(boolean → 다층 가중 최적화)는 토스가 좋아하는 "조건을 핑계가 아니라 통찰의 재료로 쓰는" 사고입니다.

**감점 사유 (-1)**
- 그러나 **QA가 문제 정의의 한 축을 반박합니다.** Agent A는 "선호 가중치(점심 직후=가능하지만 싫음)"라는 회색지대를 문제 2에서 분명히 언급했는데, **솔루션(B)이 이를 3단계로만 구현하면서 문제 정의의 절반을 잃었습니다.** 즉 문제 정의 자체는 옳았으나, **"회색지대"의 중요도를 더 강하게 못 박지 못해** 후속 단계에서 누락될 여지를 남겼습니다. (페르소나 2·5가 정확히 이 지점에서 막힘)

**보완 제안**
- 문제 정의 단계에서 "선호 가중치는 **3값(가능/선호/불가)으로 표현 불가능한 4번째 상태(회피)를 필연적으로 요구한다**"를 명제로 박아두었어야 합니다. 그래야 솔루션이 임의로 단순화하지 못합니다.

## 기준 2 — 가설 기반의 솔루션 설계

### 판정: **CONDITIONAL PASS** (7/10)

**근거 (강점)**
- Agent B는 모든 화면에 **"~게 보여주면 → ~게 인지하여 → ~게 행동할 것"** 구조를 명시했습니다. 예: *"필수를 더 크고 진한 블루로 표현하면, 학습 없이 누구를 먼저 맞춰야 하는지 직관적으로 인지할 것이다."* — 가설-검증 가능한 형태로 잘 서술됨.
- "92% 적합도 + 근거 칩"으로 **결정의 정당화 부담을 해소**하는 설계는 문제 2의 심리 허들을 정조준한 가설입니다.

**감점 사유 (-3) — QA가 핵심 가설 2개를 반박**

| 반박된 가설 | QA 증거 | 심각도 |
|---|---|---|
| **"3단계 탭 순환이면 충분하다"** (F1) | 페르소나 2·5가 "할 수 있지만 싫은 시간"을 표현 못 함 → 추천 알고리즘 신뢰도 붕괴. **문제 정의의 핵심인 '미안함 최소화'가 입력 단계에서 무너짐** | **치명적** |
| **"길게누르기 외근 블록이 직관적이다"** (F3) | 페르소나 3·6이 long-press 자체를 인지 못 함 + 부분 외근(오전만) 표현 불가 | 높음 |
| **"깔끔하면 학습 없이 쓴다"** (전체) | 페르소나 6이 진입 시 "뭘 해야 하지?" 혼란 → 온보딩 부재 | 중간 |
