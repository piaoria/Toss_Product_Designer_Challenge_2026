### 📝 에이전트 발언 내용


# 일정 조율의 본질적 문제 정의

> 일정 조율은 "시간 찾기"가 아니라 **"책임 떠넘기기와 눈치 게임"** 이다.
> 표면적으로는 빈 시간을 찾는 단순 작업처럼 보이지만, 실제로 사람들이 겪는 고통은 *조율 과정에서 발생하는 심리적 마찰*에 있다.

---

## 문제 1. 조율의 책임이 한 사람에게 쏠리고, 나머지는 '수동적 응답자'가 된다

**한 줄 요약:** 일정 조율은 주최자에게는 무한 추적 노동이고, 참석자에게는 미루고 싶은 귀찮은 알림이다.

### 왜 문제인가 (심리적·행동적 관점)
- **주최자**는 "내가 다 정리해야 한다"는 인지 부담을 진다. 카톡에서 6명의 답을 받아내려면 안 읽은 사람을 찾아 재촉하고, 흩어진 답변을 머릿속에서 교집합으로 계산해야 한다. 이건 **워킹 메모리를 갉아먹는 노동**이다.
- **참석자**는 "나 하나쯤 늦게 답해도 되겠지"라는 **책임 분산(diffusion of responsibility)** 심리에 빠진다. 6명 중 1명이라는 익명성이 응답을 미루게 만든다.
- 결과적으로 조율은 *가장 부지런한 사람이 가장 손해 보는 구조*가 되고, 이 불공정함이 협업 전체의 피로감으로 누적된다.

---

## 문제 2. '되는 시간'을 묻는 방식이 거짓 답변을 유도한다

**한 줄 요약:** 사람들은 "언제 돼?"라는 질문에 진짜 선호가 아니라 '그냥 무난한 시간'으로 대충 답한다.

### 왜 문제인가 (심리적·행동적 관점)
- "다 괜찮아요"는 협조적으로 보이려는 **사회적 바람직성 편향(social desirability bias)** 의 산물이다. 실제로는 점심 직후가 싫지만, 굳이 까다로운 사람이 되고 싶지 않아 말하지 않는다.
- 카톡 같은 텍스트 환경은 **선호의 강도(꼭 피하고 싶다 vs. 가능하면 피하고 싶다)** 를 표현할 수단이 없다. 모든 답변이 "가능/불가능"의 이진법으로 뭉개진다.
- 그래서 잡힌 회의는 '아무도 강하게 반대하지 않았지만, 아무도 만족하지 않는' 시간이 된다. 이건 **숨은 불만의 축적**이며, 회의 집중도 저하로 이어진다.

---

## 문제 3. 확정의 순간에 권위와 명확성이 사라진다

**한 줄 요약:** 누가, 언제, 무엇을 기준으로 시간을 '확정'했는지 불분명해 모두가 불안한 채로 회의 날을 맞는다.

### 왜 문제인가 (심리적·행동적 관점)
- 카톡 대화는 흐른다. "화요일 3시 어때요?"라는 메시지는 스크롤에 묻히고, **"이게 확정된 거 맞나?"** 라는 불확실성이 남는다.
- 인간은 명시적 종결(closure)이 없으면 그 일을 계속 머릿속에 띄워둔다(**자이가르닉 효과**). 즉, 확정되지 않은 회의는 *6명 모두의 인지 자원을 점유*한 채 다음 주까지 떠다닌다.
- 확정 책임자가 불분명하면 **"내가 굳이 못 박지 않아도 누군가 하겠지"** 라는 또 다른 책임 분산이 발생하고, 결국 회의 직전까지 일정이 미정으로 남는 최악의 시나리오가 반복된다.

# 핵심 통찰 (Problem Statement)

> **사용자는 "빈 시간을 찾는 도구"가 필요한 게 아니다.**
> **"내가 추적하지 않아도, 선호의 강도와 참석자의 무게가 자동으로 반영되어, 명확하게 확정되고 알림까지 끝나는" — 책임과 눈치로부터의 해방이 필요하다.**

이 정의가 솔루션 설계의 방향을 규정합니다:
- **복잡함의 단순화** → 6명의 비선호·제약·우선순위를 1개의 '추천 시간'으로 압축
- **Full-Cycle 오너십** → 응답 수집(Trigger) → 가중치 기반 추천 → 자동 확정 → 캘린더 알림까지 끊김 없이
- **논리적 설득력** → "왜 이 시간인가"를 사용자에게 납득시키는 *근거의 시각화* (예: "필수 4명 전원 가능, 비선호 0건")

다음 단계로 이 문제 정의를 기반으로 한 **솔루션 설계와 UI 플로우**를 제안드릴 수 있습니다.

# 일정 조율 UX 설계 — "추적하지 않아도 끝나는 경험"

> Problem Framer의 정의를 그대로 솔루션의 척추로 삼습니다.

## 1. Full-Cycle 유저 플로우

```
[TRIGGER]
주최자: "다음 주, 6명, 1시간 회의" 생성
   │  └─ 필수/선택 참석자 지정 (드래그로 두 그룹 분리)
   │  └─ 시스템이 캘린더 연동 → 빈 슬롯 자동 스캔
   ▼
[제안 — 시스템이 먼저 일함]
시스템: "후보 시간 3개"를 추천 (빈칸이 아닌 '이미 채워진 답'을 제시)
   │  └─ 각 후보에 근거 태그: "필수 4명 OK · 비선호 0"
   ▼
[참여자 응답 — 눈치 게임 제거]
참석자: 푸시 알림 → 한 화면에서 후보 3개에 대해
   │  ┌─ ◎ 좋아요 / ○ 가능 / △ 가능하지만 별로 / ✕ 불가
   │  └─ "반복 제약 저장" 토글 (외근 요일 → 다음부터 자동 반영)
   │  
   │  [미응답자 자동 처리]
   │  └─ 마감 12시간 전 미응답 → 개인 리마인드
   │  └─ 마감 시 미응답 = '시스템이 캘린더로 추정' (불확실 표시)
   ▼
[확정 — 시스템이 책임을 가져감]
시스템: 가중치 계산 → 최적 1개 자동 하이라이트
   │  └─ 주최자는 '확정' 1탭 (or 마감 시 자동 확정)
   │  └─ 확정 근거 카드 동봉: "왜 이 시간인가"
   ▼
[알림 — 종결의 못 박기]
전원 캘린더 자동 등록 + 확정 카드 푸시
   └─ "수요일 14시 확정 · 6명 전원 일정 등록됨" (자이가르닉 해소)
```

핵심 설계 원칙: **빈 화면에서 시작하지 않는다.** 시스템이 먼저 후보를 제시해 응답을 '선택'으로 축소 → 책임 부담을 8할 줄인다.

## 2. 핵심 화면 3선 + 설계 의도

### 화면 ① 참석자 응답 화면 (가장 많이 쓰임)

**UI 구성**
- 상단: "다음 주 회의 · 1시간 · 응답 4/6명 완료" (실시간 진행 바)
- 후보 카드 3개 세로 배열. 각 카드에 4단계 강도 슬라이더형 버튼:
  `😄 좋아요  🙂 가능  😕 별로  🚫 안돼요`
- 점심 직후 슬롯은 카드 우측에 옅은 회색 `🍽 점심 직후` 보조 라벨 (강요 아닌 환기)
- 하단 고정: "이 요일 매주 외근이에요" 토글 → 반복 제약 저장

> **설계 의도:** 후보를 3개로 좁혀 카드로 보여주면, 사용자는 '빈 시간을 떠올리는 노동' 없이 '평가만 하면 된다'고 인지하여, 30초 안에 응답을 끝낼 것이다.
> **설계 의도:** 4단계 강도 버튼을 색·표정으로 보여주면, 사용자는 '별로'를 누르는 게 까다로운 사람이 되는 게 아니라 정상 선택지라고 인지하여, 사회적 바람직성 편향 없이 솔직하게 답할 것이다.

---

### 화면 ② 주최자 대시보드 — '추적 노동의 자동화'

**UI 구성**
- 상단: 추천 1순위 시간이 큰 카드로 강조 + 근거 칩
  `필수 전원 OK · 비선호 0건 · 미응답 1명(캘린더상 가능)`
- 그 아래 후보 2·3순위는 작게 (점수순)
- 참석자 응답 현황: 아바타 6개, 응답 완료=컬러 / 미응답=점선 + "콕 찌르기" 버튼
- '콕 찌르기'는 주최자가 직접 재촉하는 게 아니라 시스템 알림으로 발송

> **설계 의도:** 흩어진 6명의 답을 시스템이 '점수 + 근거'로 압축해 보여주면, 주최자는 머릿속 교집합 계산 노동에서 해방되어, 추적이 아닌 '확인 후 확정'만 하면 된다고 인지할 것이다.
> **설계 의도:** 재촉을 시스템 발송 '콕 찌르기'로 대신하면, 주최자는 인간관계 마찰 없이 행동할 수 있다고 인지하여, 미루지 않고 즉시 리마인드할 것이다.

---

### 화면 ③ 확정 카드 — '못 박기와 종결'

**UI 구성**
- 큰 시간 표기: **수요일 14:00–15:00 ✓ 확정**
- "왜 이 시간인가" 근거 펼침:
  - ✅ 필수 참석자 4명 전원 가능
  - ✅ 강한 비선호 0건
  - ⚠️ 선택 참석자 1명만 '별로' (대안 없어 불가피)
- 하단: "전원 캘린더에 등록되었습니다" + 변경 요청 버튼(작게)

> **설계 의도:** 확정 시간을 근거와 함께 단일 카드로 못 박아 보여주면, 사용자는 '이건 진짜 정해졌다'는 명시적 종결을 인지하여, 회의 날까지 일정을 머릿속에 띄워두지 않을 것이다(자이가르닉 해소).

## 3. 필수/선택 참석자 가중치의 시각화

가중치는 **숫자가 아니라 위계와 색의 차이**로 직관 표현합니다.

### (1) 응답 현황 — 아바타의 시각적 위계
```
필수 참석자  ●●●●        ← 크게, 진한 컬러 링, 상단 줄
선택 참석자  ◦◦          ← 작게, 옅은 그레이, 하단 줄, "선택" 라벨
```
- 필수 1명이 '안돼요'면 → 후보 카드 전체가 **빨간 경고 테두리** ("필수 1명 불가")
- 선택 1명이 '안돼요'면 → 작은 회색 각주 ("선택 1명 불가") — 추천에서 거의 감점 안 함

### (2) 추천 점수 바 — 무게가 다른 막대
각 후보 카드의 점수 바에서 필수/선택 응답을 **분리된 두 층**으로 표시:
```
후보 A   [필수층 ████████ 100%] [선택층 ▓▓▓░░ 60%]
                 ↑ 굵고 진함        ↑ 가늘고 옅음
```
- 필수층은 **굵고 진한 막대**, 선택층은 **가늘고 흐린 막대**로 시각적 무게 자체를 다르게 → "필수가 더 중요하다"를 학습 없이 인지.

### (3) 확정 근거 카드 — 위계 순서로 나열
근거를 항상 **필수 먼저, 선택 나중** 순서로 배치하고, 필수 조건은 ✅ 굵게 / 선택 조건은 작은 회색으로.

> **설계 의도:** 가중치를 '크기·진하기·위치'라는 3중 시각 위계로 표현하면, 사용자는 별도 설명 없이도 "위/굵은/진한 것이 더 중요하다"는 보편적 직관으로 우선순위를 즉시 이해하여, 잘못된 다수결 합의(필수 불가 시간 강행)를 스스로 피할 것이다.

### 평가: **PASS** / **9점**

**근거 (왜 통과인가)**
