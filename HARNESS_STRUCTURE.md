# 🏢 토스 기획 과제 — 에이전트 하네스 구조 설계 문서

---

## 왜 4개 에이전트인가?

토스의 기획 평가 기준은 세 가지입니다:
**본질적 문제 정의 → 가설 기반 솔루션 → 조형적 완성도**

이 3단계 사고를 한 에이전트가 전부 처리하면 각 단계의 깊이가 얕아집니다.
역할 페르소나가 명확할수록 더 날카로운 관점이 나오기 때문에,
각 단계를 전담하는 에이전트 + 전체를 검증하는 에이전트로 나눴습니다.

```
Orchestrator
├── Agent A: Problem Framer     → 문제 정의 단계
├── Agent B: UX Designer        → 솔루션 설계 단계
├── Agent C: Visual Critic      → 시각 완성도 단계
└── Agent D: Toss Evaluator     → 3기준 교차 검증
```

---

## 에이전트별 설계 의도

---

### 🔍 Agent A: Problem Framer
**역할:** UX 리서처 / 문제 정의 전문가
**실제 팀 비유:** 킥오프 미팅에서 "우리가 진짜 풀어야 할 게 뭔지" 먼저 정리하는 사람

**왜 이 에이전트가 첫 번째인가?**

기획의 첫 단추는 "무엇을 만들 것인가"가 아니라 "무엇이 진짜 문제인가"입니다.
일정 조율이 어렵다는 건 표면적 현상일 뿐입니다.
그 이면에 있는 심리적·행동적 허들 — 인지 과부하, 눈치 보기, 조율 책임의 분산 —
을 먼저 정의하지 않으면, 아무리 좋은 UI를 만들어도 핵심을 빗나갑니다.

뒤에 오는 모든 에이전트의 작업 방향이 이 에이전트의 문제 정의에서 출발합니다.
순서상 첫 번째인 이유가 여기에 있습니다.

**컨텍스트 전달:** 과제 조건만 받음 (이전 에이전트 없음)

---

### 🎨 Agent B: UX Designer
**역할:** 프로덕트 디자이너 / 인터랙션 설계자
**실제 팀 비유:** 화이트보드 앞에서 유저 플로우를 그리고 각 화면의 의도를 설명하는 사람

**왜 이 에이전트가 두 번째인가?**

문제가 정의된 다음엔 "어떻게 풀 것인가"를 설계해야 합니다.
여기서 중요한 건 기능 나열이 아니라 가설 기반 UX 설계입니다.
"~게 보여주면 사용자는 ~게 인지하여 ~게 행동할 것이다"라는
명확한 설계 의도와 근거가 있어야 토스 수준의 기획이 됩니다.

Problem Framer의 문제 정의를 인풋으로 받아,
그 문제를 직접 해결하는 플로우와 화면을 설계합니다.
문제와 해결이 1:1로 연결되어야 가설이 설득력을 가집니다.

**컨텍스트 전달:** 과제 조건 + Agent A 결과물

---

### ✏️ Agent C: Visual Critic
**역할:** UI 스페셜리스트 / 비주얼 디테일 검증자
**실제 팀 비유:** 디자인 리뷰에서 "이 색이 여기서 맞나?", "이 상태 표현이 명확한가?" 짚어주는 사람

**왜 이 에이전트가 세 번째인가?**

토스의 평가 기준 3번은 "흔한 캘린더 UI도 이 맥락에 맞게 픽셀 단위로 다듬어야 한다"입니다.
좋은 UX 플로우가 있어도, 실제 화면에서 컬러·타이포·간격·컴포넌트 상태가
이 특수한 맥락(필수/선택, 외근, 기피 시간대)을 반영하지 않으면 완성도가 낮아집니다.

UX Designer의 설계가 실제 화면으로 구현될 때
어떤 시각적 결정이 필요한지 구체적으로 채워주는 역할입니다.

**컨텍스트 전달:** 과제 조건 + Agent A + Agent B 결과물

---

### ⚖️ Agent D: Toss Evaluator
**역할:** 기획 검증자 / 토스 기준 평가자
**실제 팀 비유:** 최종 발표 전 리드 디자이너가 "토스 기준으로 이게 통과할 수 있나?" 냉정하게 체크하는 사람

**왜 이 에이전트가 마지막인가?**

세 에이전트가 각자의 관점에서 작업한 결과물을,
토스가 실제로 평가하는 3가지 기준에 비춰 교차 검증합니다.
이 단계 없이 제출하면 좋은 아이디어도 평가 기준을 빗나갈 수 있습니다.

통과 여부뿐 아니라 보완점도 구체적으로 짚어주기 때문에,
이 에이전트의 피드백을 반영해 기획서를 고도화할 수 있습니다.

**컨텍스트 전달:** 과제 조건 + Agent A + B + C 전체 결과물

---

## 실행 흐름 요약

```
[과제 조건 입력]
        ↓
  🔍 Problem Framer
  → 본질 문제 3가지 정의
        ↓
  🎨 UX Designer
  → 플로우 + 핵심 화면 3개 설계
        ↓
  ✏️ Visual Critic
  → 컬러/타이포/컴포넌트 디테일 제안
        ↓
  ⚖️ Toss Evaluator
  → 3기준 Pass/Fail + 점수 + 보완 제안
        ↓
  [회의록 MD 자동 저장]
```

각 에이전트는 이전 에이전트의 결과물을 누적으로 받습니다.
Toss Evaluator는 A+B+C 전체를 보고 종합 판단합니다.

---

## 토큰 비용 구조

에이전트가 뒤로 갈수록 컨텍스트가 누적되어 입력 토큰이 증가합니다.

| 에이전트 | 입력 규모 | 이유 |
|---------|---------|------|
| Problem Framer | 작음 | 과제 조건만 |
| UX Designer | 중간 | 조건 + A 결과 |
| Visual Critic | 큼 | 조건 + A + B 결과 |
| Toss Evaluator | 가장 큼 | 조건 + A + B + C 전체 |

> 비용 최적화 팁: Visual Critic과 UX Designer는 병렬 실행 가능하나,
> 현재 구조는 순차 실행으로 각 결과가 다음 에이전트의 인풋이 됩니다.
